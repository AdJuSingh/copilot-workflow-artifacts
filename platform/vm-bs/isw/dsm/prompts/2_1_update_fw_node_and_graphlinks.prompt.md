# Complete Diagnostic Database Update

## What This Does

### Input
- **Three Text Files**: Structured key:value pairs for complete diagnostic database integration
  - **FAILURE_WORD File**: `Diamant_Generate_FWSplitter.txt` - Failure word definitions
  - **NODE File**: `Diamant_Generate_NodeSplitter.txt` - Node definitions with FW references
  - **Graph Link File**: `Diamant_Generate_GraphLinks.txt` - Node hierarchy relationships
  - Search for files in `GenFromCoPilot\` directory

### Process
1. Loads existing FAILURE_WORD, NODE, and NODE_LINK databases from selected configuration
2. Parses all three input files to extract entry specifications
3. Creates or updates FAILURE_WORD entries with properties, debouncers, monitoring
4. Creates or updates NODE entries with FW_REF linkages
5. Creates NODE_LINK entries to establish parent-child node relationships
6. Persists all changes to XML files via automatic routing

### Output
- **Updated XML Files**: 
  - FAILURE_WORD files in `rb\as\ms\core\app\<sw_group>\RBAPLCust\cfg\`
  - NODE files based on KEYLIST routing
  - Graph Link files in `rb\as\mb\<product_line>\dsmpr\cfg\Diamant_Cfg\`
- **Log File**: `Diamant_Update_FailureNodeGraph_CLI_Log.txt` with complete execution trace
- **Console Summary**: Created/updated counts for all three entity types, affected files

### Key Capabilities
- **All-in-One Processing**: Complete diagnostic database setup in single execution
- **Optional Input Files**: Provide 1, 2, or all 3 input files as needed
- **Dependency-Aware**: Processes FW first, then NODEs, then Graph Links
- **Duplicate Detection**: Checks existing entries before creating new ones
- **Automatic Routing**: Determines correct XML file locations based on KEYLIST values

## Workflow

When performing complete diagnostic database update, follow these steps:

1. **Show Available Configurations**
   - List all folders in `Gen/` directory
   - Display as numbered options for selection

2. **Select Configuration**
   - Wait for my configuration choice
   - Use selected config to construct command

3. **Verify Input Files**
   - Search for all three input files in `GenFromCoPilot\splitter_input\` directory:
     - `Diamant_Generate_FWSplitter.txt`
     - `Diamant_Generate_NodeSplitter.txt`
     - `Diamant_Generate_GraphLinks.txt`
   - Report which files are found
   - At least one file must exist to proceed

4. **Display Input File Contents**
   - Read and show key information from each found file:
     - **FW File**: Global config (Customer, SW_GROUP_REF, PRODUCT_LINE_REF), entry count
     - **NODE File**: Global config, node names, FW references
     - **Graph Link File**: Link count, source/target pairs
   - Validate KEYLIST consistency across files

5. **Execute CLI Command**
   - Construct command with environment variables for all found files
   - Run in PowerShell terminal
   - Monitor execution output

6. **Verify Results**
   - Check console for success indicators:
     - "Parsed X failure word entries" (FW input parsed)
     - "FAILURE_WORD processing completed! Created: X, Updated: Y"

     - ✅ FAILURE_WORD Results: Created/Updated counts
     - ✅ NODE Results: Created/Updated counts
     - ✅ Graph Link Results: Created/Existing counts
   - Show list of modified XML files with full paths

7. **Track Modified Files for Review**
   - Extract all XML file paths from "Splitter [...] written" messages in console output
   - Create or update tracking file: `GenFromCoPilot\Review\Failure_Node_Graph_filesToReview.txt`

### Error Handling
- No input files found → report missing files, request correct paths
- Configuration CSV missing → check Gen/<Config>/make/ directory
- Database load failed → verify READSPLITTER parameter
- Invalid input format → check for required keys (Failure Name, Node name, Source/Target)
- KEYLIST inconsistency → warn about mismatched Customer/SW_GROUP_REF values
- File not found → verify absolute paths, use Test-Path
- Exit code 1 with success messages → **Normal** (unrelated plugin errors)

### Input File Validation Checklist
Before execution, verify:
- ✅ FW file has `Failure Name:` for each entry
- ✅ NODE file has `Node name:` for each entry
- ✅ Graph Link file has both source and target for each link
- ✅ Global config values are consistent across files
- ✅ SW_GROUP_REF values match (accounting for SwGrp_ prefix)
- ✅ NODE references FWs that are defined in FW file
- ✅ Graph Links reference NODEs that are defined in NODE file

## Use Cases

### Use Case 1: Complete New Feature Integration
When adding a new diagnostic feature that requires failure definitions, node structure, and hierarchy:
```
1. Create all three input files with related entries
2. Execute complete update to establish all relationships in one pass
3. Verify FW → NODE → Graph Link chain is complete
```

### Use Case 2: Add Graph Links Only (FW and NODE Already Exist)
When failure words and nodes are already in database, but hierarchy needs to be established:
```
1. Provide only DIAMANT_GRAPHLINK_INPUT_FILE
2. Execute to create parent-child relationships
3. Verify NODE_LINK entries created
```

### Use Case 3: Incremental Update (Add NODEs and Links to Existing FWs)
When failure words exist, but new nodes need to be added:
```
1. Provide DIAMANT_NODE_INPUT_FILE and DIAMANT_GRAPHLINK_INPUT_FILE
2. Skip FW processing (already exists)
3. Create NODEs linking to existing FWs
4. Establish node hierarchy
```
