# Create Review Report

## Purpose
Generate a comprehensive review report by verifying Diamant XML configuration files against DEM requirements documentation.

## Context
This prompt is used to create structured review reports that validate Diamant XML configurations against documented requirements. The XML files are generated by executing DiamantPro plugins (.dpp files) in three main workflow stages:
1. **Stage 2.1.2**: Failure Word, Node, and Graph Links Update
2. **Stage 2.1.3**: DTC Complete Workflow Update  
3. **Stage 2.5**: STM Configuration Update

Each workflow stage generates tracking files that list all modified XML files.

## Input Files
1. **Requirements Documentation**: `GenFromCoPilot\requirements\DEM_Requirements.md` - Complete DEM requirements and specifications
2. **Workflow Tracking Files** (auto-generated in `GenFromCoPilot\Review\`):
   - `Failure_Node_Graph_filesToReview.txt` - FW, NODE, and GraphLink XML files from Stage 2.1.2
   - `DTC_filesToReview.txt` - DTC XML files from Stage 2.1.3
   - `STM_filesToReview.txt` - STM configuration files from Stage 2.5
   
   Each tracking file contains exact paths to all modified Diamant XML files with execution metadata.

## Instructions

### Step 1: Review Input Preparation
1. **Read DEM Requirements**: Parse `GenFromCoPilot\requirements\DEM_Requirements.md` to understand all requirements, specifications, and expected configurations
2. **Scan Tracking Files**: List all `.txt` files in `GenFromCoPilot\Review\` directory
3. **Verify Workflow Completion**: Check that all expected tracking files exist:
   - `Failure_Node_Graph_filesToReview.txt` (Stage 2.1.2)
   - `DTC_filesToReview.txt` (Stage 2.1.3)
   - `STM_filesToReview.txt` (Stage 2.5)
4. **Parse Tracking Files**: Extract all XML file paths listed in each tracking file
5. If any tracking file is missing, inform user which workflow stage hasn't been executed and abort review

### Step 2: Locate and Parse Diamant XML Files
Based on tracking files in `GenFromCoPilot\Review\`, extract and verify all XML file paths:

**From Failure_Node_Graph_filesToReview.txt:**
- Extract all FAILURE_WORD XML file paths (Diamant__FW__*.xml)
- Extract all NODE XML file paths (Diamant__NODE__*.xml)
- Extract all Graph Link XML file paths (Diamant__GraphLink__*.xml)
- Note execution timestamps and configuration names

**From DTC_filesToReview.txt:**
- Extract all DTC XML file paths (Diamant__DTC__*.xml)
- Extract all DTC Attribute XML file paths (Diamant__DTCATTRIBUTES__*.xml, Diamant__DTC_AttributesClassLink__*.xml)
- Extract all FW-DTC Link file paths (Diamant__DTCLINK__PROJECT.xml)
- Note execution timestamps and processing statistics

**From STM_filesToReview.txt:**
- Extract all STM XML file paths (Diamant__STM__PROJECT*.xml)
- Note which nodes were updated/added
- Note which STM_PROJECTLINKs were modified

**Verify File Existence:**
- Check that all extracted file paths actually exist
- Report any missing files as critical issues

**Identify Referenced/Pre-Existing Entities (Not Newly Created):**
- **Critical Principle:** Any entity involved in the configuration must be verified, whether newly created OR pre-existing
- Review tracking files for indicators of pre-existing entities:
  - "already exists" - entity was found in database and reused
  - "0 created" or "0 added" - no new entities of this type created
  - "reused existing" - explicit statement that entity pre-dates this workflow
- Search for definition files containing pre-existing entities:
  - **Failure Words:** If tracking shows FW already exists, search for Diamant__FW__*.xml files containing it
  - **Nodes:** If tracking shows node already exists, search for Diamant__NODE__*.xml files containing it
  - **DTCs:** If DTC workflow tracking shows "0 DTCs created" but DTC is referenced, search for Diamant__DTC__*.xml
  - **DTC Attributes:** If tracking shows "already exists" or "reused", search for Diamant__DTCATTRIBUTES__*.xml or Diamant__DTC_Attribute__*.xml
  - **STM Components:** If STM tracking mentions pre-existing state machines, search for relevant STM__PROJECT files
- Use grep_search or file_search to locate definition files
- **Add ALL definition files to review scope**, regardless of whether they appear in tracking files
- Document for each pre-existing entity:
  - Entity name
  - File path where defined
  - Whether it appears in tracking files or was discovered through search

### Step 3: Extract Configuration Data from XML
**For each Diamant XML file found (including pre-existing entity definition files), extract the following information:**

**IMPORTANT:** Extract configuration for BOTH newly created AND pre-existing entities. If a workflow reuses an existing entity (e.g., "DTCAttribute_CHR_NonOBD already exists"), you must still extract and verify its configuration.

**From FAILURE_WORD XML:**
- Failure word names (e.g., FW_HydraulicUndervoltage1)
- KEYLIST values (Customer, SW_GROUP_REF, PRODUCT_LINE_REF)
- DESCRIPTION, ROOTCAUSE, ALGORITHM_DESCRIPTION, GOOD_CHECK_DESCRIPTION
- Debouncer configuration (type, task cycle, detection time, recovery time, thresholds)
- DTC_REF (DTC code linkage)
- OPERATION_CYCLE, DTC_SEVERITY, WARNING_BIT_INDICATOR
- UMID (if present)

**From NODE XML:**
- Node names (e.g., Node_VoltageHandler1)
- KEYLIST values
- PARENT_NODE_REF (parent-child relationships)
- FW_REF entries (failure words assigned to node)
- PRIORITY values
- SYSTEM_STRATEGY

**From GraphLink XML:**
- NODE_LINK entries
- SOURCE_NODE and TARGET_NODE pairs (parent-child graph structure)

**From DTC XML:**
- DTC codes (e.g., 0x123456)
- DTC descriptions
- OPERATION_CYCLE
- DTC_ATTRIBUTE_REF

**From DTC_ATTRIBUTE XML:**
- DTC attribute names (e.g., DTCAttribute_CHR_NonOBD)
- KEYLIST values (Customer, Project, Software Group, Product Line)
- DTC Kind (DTC_KIND_ALL_DTCS, etc.)
- Freeze frame class (DemFreezeFrameClassRef)
- Extended data class (DemExtendedDataClassRef)
- Number of freeze frame records (DemMaxNumberFreezeFrameRecords)
- Failure cycle counter threshold (EventFailurecyclecounterThreshold)
- Aging cycle counter threshold (AgingcyclecounterThreshold)
- DTC event destination (EventDestination: Primary/Secondary memory)
- DTC Priority (DemDTCPriority)
- AgingcycleRef (operation cycle for aging)
- **IMPORTANT:** Even if DTC attribute file is not in tracking files, if it's referenced by a DTC, it MUST be verified against requirements

**From STM__PROJECT XML:**
- STM_PROJECTLINK entries (STM component names)
- STM_NODE entries within each project link
- NODE_REF (which nodes are degraded)
- STATE_REF (to which states nodes are degraded)

### Step 4: Cross-Reference with DEM Requirements
Map extracted XML data to requirements documented in `DEM_Requirements.md`:

**Requirement Verification Categories:**

**A. Failure Word Requirements Verification**
- Compare failure word names in XML with those specified in requirements
- **Verify ALL failure words mentioned in requirements**, whether newly created or pre-existing:
  - If workflow creates new FW: Verify against requirements
  - If workflow references existing FW: Locate FW definition file and verify against requirements
  - Example: If requirements specify FW_HydraulicUndervoltage1 but it already existed in database, still verify its configuration matches requirements
- Verify KEYLIST values (Customer, SW_GROUP_REF, PRODUCT_LINE_REF) match requirements
- Check DESCRIPTION, ROOTCAUSE, ALGORITHM_DESCRIPTION completeness
- Verify debouncer configuration (type, thresholds, timing) matches specification
- Validate DTC_REF assignments match requirement specifications
- Check OPERATION_CYCLE, DTC_SEVERITY settings per requirements
- Verify UMID presence where required by specifications
- **Mark as ❌ FAIL** if pre-existing FW parameters don't match requirements

**B. Node Configuration Requirements Verification**
- Verify all required nodes are present in NODE XML files
- **Verify ALL nodes mentioned in requirements**, whether newly created or pre-existing:
  - If workflow creates new node: Verify against requirements
  - If workflow modifies existing node (e.g., adding FW_REFs): Verify entire node configuration matches requirements
  - Example: If adding FW to existing Node_VoltageHandler1, verify all node properties (Customer, SW Group, priorities, etc.) match requirements
- Check node naming conventions match requirements
- Validate FW_REF assignments (failure words to nodes) per specifications
- Verify PRIORITY values match documented requirements
- Check parent-child node relationships in GraphLink XML
- Validate SYSTEM_STRATEGY configurations
- **Mark as ❌ FAIL** if pre-existing node parameters don't match requirements

**C. DTC Configuration Requirements Verification**
- Verify all required DTCs are present in DTC XML files
- **Verify ALL DTCs mentioned in requirements**, whether newly created or pre-existing:
  - If workflow creates new DTC: Verify against requirements
  - If workflow references existing DTC: Locate DTC definition file and verify against requirements
- Check DTC codes match requirement specifications (0xXXXXXX format)
- Validate DTC descriptions match documented requirements
- Verify DTC_ATTRIBUTE_REF assignments are correct
- **Verify Referenced/Pre-Existing DTC Attributes (Critical - applies to all existing entities):**
  - If DTC links to an existing attribute (e.g., "DTCAttribute_CHR_NonOBD already exists"), locate the attribute definition file
  - Extract attribute configuration parameters from definition file
  - Compare attribute parameters against DEM_Requirements.md specifications:
    - Customer, Product Line, Software Group (if specified)
    - Freeze frame class and extended data class
    - Number of freeze frame records
    - Failure cycle counter threshold
    - Aging cycle counter threshold
    - DTC event destination
    - DTC Priority
  - **Mark as ❌ FAIL** if required attribute parameters don't match requirements
  - Mark as ⚠️ PARTIAL if attribute exists but some parameters missing from requirements (not verifiable)
- Check OPERATION_CYCLE consistency across related entities
- Validate freeze frame and extended data configurations
- Verify aging and failure counter thresholds per requirements
- **Mark as ❌ FAIL** if pre-existing DTC parameters don't match requirements

**D. STM Configuration Requirements Verification**
- Verify all required nodes are configured in STM_PROJECT XML
- **Verify ALL STM components mentioned in requirements**, whether newly created or pre-existing:
  - If workflow creates new STM_NODE entries: Verify against requirements
  - If workflow references existing STM components: Verify entire STM_PROJECTLINK configuration matches requirements
- Check STM_PROJECTLINK entries match requirement specifications
  - **CRITICAL:** If DEM_Requirements.md specifies a state machine (e.g., "TargetStateHBA = Off"), it MUST be configured in XML
  - If state machine is not found in database during execution, this is a **MISSING CONFIGURATION**, not an acceptable skip
  - Mark as ❌ FAIL if required state machine missing from XML, even if tracking file says "skipped" or "not found"
- Validate node degradation states (STATE_REF values) per requirements
- Verify delay_degradation settings are correct
- Check completeness of STM configuration coverage
  - Count: Total state machines specified in requirements vs. actually configured in XML
  - Report any discrepancies as missing configurations
- **Mark as ❌ FAIL** if pre-existing STM component parameters don't match requirements

**E. Cross-Entity Consistency Verification**
- Verify FW → DTC → Node → STM relationships are complete and consistent
- Check that all failure words have corresponding DTC references
- Verify all nodes with failure words have STM degradation configurations
- Validate graph link parent-child relationships are complete
- Check for orphaned or unreferenced entities

### Step 5: Identify Issues and Generate Findings
Compare XML configurations against DEM_Requirements.md and categorize findings by severity:

- **Critical**: 
  - Required configurations missing from XML
  - Configuration values contradict requirements
  - Missing STM configuration for critical nodes
  - Invalid DTC links or missing DTCs
  - Inconsistent operation cycles across related entities
  - Missing parent nodes in graph structure

- **High**: 
  - Required descriptions missing or incomplete
  - Configuration values not matching requirement specifications
  - Missing UMID where required
  - Invalid or inconsistent priority values
  - Incomplete freeze frame or extended data configuration
  - **Missing required STM configurations** (state machine specified in requirements but not configured in XML)
  - Required DTC attributes missing or not linked
  - **Pre-existing entity parameters not matching requirements** (applies to ALL entity types: FW, Node, DTC, DTC Attribute, STM)
    - Example: Existing DTCAttribute has 3 freeze frames but requirements specify 2
    - Example: Existing FW has Failed threshold 50 but requirements specify 100
    - Example: Existing Node has wrong Customer or SW Group

- **Medium**: 
  - Optional fields missing but recommended
  - Generic descriptions where specific ones expected
  - Minor inconsistencies in naming conventions
  - Missing optional attributes

- **Low**: 
  - Formatting issues
  - Documentation gaps
  - Suggested improvements for maintainability

### Step 6: Generate Comprehensive Report
Create a structured markdown report with:

1. **Executive Summary**
   - Overall status:
     - ✅ PASS if all requirements met (100% compliance, 0 critical/high issues)
     - ⚠️ PARTIAL PASS if some requirements missing (90-99% compliance, OR 1+ high issues but 0 critical)
     - ❌ FAIL if critical requirements missing (<90% compliance, OR 1+ critical issues)
   - Total requirements vs. verified configurations
   - Critical/High/Medium/Low issue counts
   - Workflow stage completion status
   - Key findings and strengths
   - **Explicitly state which requirements are missing if partial pass or fail**

2. **Workflow Execution Summary**
   - Stage 2.1.2 (FW/Node/Graph): Files modified, execution timestamp, statistics
   - Stage 2.1.3 (DTC): Files modified, execution timestamp, statistics
   - Stage 2.5 (STM): Files modified, execution timestamp, statistics

3. **Requirements Verification by Category**
   - For each requirement category (A-E):
     - Requirement description from DEM_Requirements.md (quote specific requirement text)
     - Status:
       - ✅ **Pass**: All requirements in this category fully implemented
       - ⚠️ **Partial**: Some requirements missing or incomplete (specify which ones)
       - ❌ **Fail**: Critical requirements not met (specify which ones)
     - Evidence from XML files (with file paths and line references)
       - **Include evidence from ALL relevant files** (newly created AND pre-existing entity definition files)
       - For each entity in requirements, document whether it was newly created or pre-existing
     - Comparison table: Expected (from DEM_Requirements.md) vs. Actual (from XML)
       - For missing configurations, clearly show "Expected: [value from requirements]" vs "Actual: MISSING"
       - **For pre-existing entities:** Show "Expected: [value from requirements]" vs "Actual: [value from definition file]" vs "Status: Match/Mismatch"
       - Clearly label each row as "New" or "Pre-existing" for traceability
     - Issues identified (with severity: Critical/High/Medium/Low)
     - Recommendations for resolving each issue

4. **Configuration Validation Summary**
   - Table of validated elements (KEYLIST values, DTC mappings, node structure, STM configurations)
     - Include column indicating "New" or "Pre-existing" for each element
   - Table of requirements vs. actual implementation
   - Table of missing/unverified configurations
   - Table of pre-existing entity parameter mismatches (if any)
   - Cross-entity relationship validation results

5. **Requirements Traceability Matrix**
   - Requirement ID → XML File → Configuration Element
   - Status of each requirement:
     - ✅ **Verified**: Requirement fully implemented in XML
     - ⚠️ **Partial**: Requirement partially implemented (document what's missing)
     - ❌ **Not Implemented**: Requirement completely missing from XML
   - For missing configurations, document:
     - Requirement text from DEM_Requirements.md
     - Expected configuration
     - Actual status (e.g., "N/A - State machine not found in database")
     - Root cause if known
     - Required action to resolve

6. **Risk Assessment**
   - Overall risk level:
     - 🟢 **LOW**: All requirements met, 0 critical/high issues
     - 🟡 **MEDIUM**: 1-2 high issues (missing non-critical configurations), 95%+ compliance
     - 🟠 **HIGH**: 3+ high issues or 1 critical issue, 90-95% compliance
     - 🔴 **CRITICAL**: Multiple critical issues or <90% compliance - release blocker
   - Critical risks (requirements not met - release blockers)
     - Missing core configurations (DTC, Node, critical failure words)
   - High priority items (deviations from requirements - pre-release required)
     - Missing required STM configurations
     - Missing required DTC attributes or links
   - Medium priority items (optional requirements not met)
   - Mitigation recommendations with ownership and timelines

7. **Recommendations**
   - Immediate actions (HIGH PRIORITY - must resolve before production):
     - Configurations contradicting requirements (wrong values)
     - **Missing required configurations** (specified in requirements but not in XML)
       - For each missing config: provide specific action (e.g., "Add TargetStateHBA state machine to database and re-execute STM workflow")
       - Assign owner (e.g., "System Architect + DEM Configuration Engineer")
       - Set timeline (e.g., "Within 3 business days")
   - Short-term actions (missing required configurations, pre-production required)
   - Follow-up actions (optional improvements)
   - Configuration updates needed

8. **Metrics**
   - Requirements compliance percentage:
     - Formula: (Requirements Met / Total Requirements) × 100
     - Include breakdown: X passed, Y partial, Z failed
     - **Count each missing configuration as a failed requirement**
       - Example: If requirements specify 4 STM state machines but only 3 configured, compliance = 75% for that category
     - **Count pre-existing entity parameter mismatches as failed requirements**
       - Example: If existing DTCAttribute has wrong freeze frame count, mark as failed
   - Issue distribution by severity (Critical/High/Medium/Low counts)
   - Configuration coverage by category:
     - Failure Words: X/Y configured (Z%) - include breakdown of new vs. pre-existing
     - Nodes: X/Y configured (Z%) - include breakdown of new vs. pre-existing
     - DTCs: X/Y configured (Z%) - include breakdown of new vs. pre-existing
     - STM Configurations: X/Y configured (Z%) ← This is where missing TargetStateHBA would show
   - Files reviewed per workflow stage
     - Include count of additional files reviewed (pre-existing entity definition files)

9. **Appendices**
   - Reference documents:
     - DEM_Requirements.md (requirements source)
     - Tracking files used (with timestamps)
   - Complete list of XML files reviewed (by workflow stage)
     - Separate section for "Additional Files Reviewed" (pre-existing entity definition files not in tracking)
   - Configuration summary tables:
     - Failure Words configured (with "New"/"Pre-existing" indicator)
     - Nodes configured (with "New"/"Pre-existing" indicator)
     - DTCs configured (with "New"/"Pre-existing" indicator)
     - STM PROJECTLINKs configured (with "New"/"Modified" indicator)
   - Review metadata (date, configuration, reviewer)

## Output Format
The report should be saved as:
- **File**: `GenFromCoPilot\Review\DEM_Review_Report.md`
- **Format**: Markdown with proper headings, tables, and code blocks
- **Sections**: Header, Executive Summary, Detailed Findings, Validation Summary, Risk Assessment, Recommendations, Metrics, Appendices

## Example Usage
```
Review the DEM configuration by analyzing all Diamant XML files tracked in

```

## Notes
- **Tracking Files**: Read ALL `.txt` files in `GenFromCoPilot\Review\` directory - don't hardcode filenames
- **Requirements Source**: Use `DEM_Requirements.md` as the single source of truth for verification
- **XML Parsing**: Use grep search or file read to extract XML element values
- **File Paths**: Always include full file paths in evidence sections with markdown links
- **Line References**: When citing XML content, include approximate line numbers if available
- **Status Indicators**: Use ✅ (verified) ⚠️ (partial) ❌ (fail) for visual clarity
- **Tables**: Format findings in tables for easy scanning
- **Critical Focus**: 
  - Requirements compliance (does XML match DEM_Requirements.md?)
  - Cross-entity consistency (FW → DTC → Node → STM relationships)
  - Completeness (all required entities present)
- **Evidence-Based**: For each finding, cite specific requirement from DEM_Requirements.md and corresponding XML configuration
- **Traceability**: Map each requirement to specific XML file and configuration element
- **Missing vs. Wrong**: Clearly distinguish between "configuration missing" vs "configuration wrong" vs "not in requirements"
- **Status Determination Rules**:
  - ✅ **PASS**: All requirements from DEM_Requirements.md are implemented correctly in XML (100% compliance)
  - ⚠️ **PARTIAL PASS**: Some requirements missing or partially implemented (typically 90-99% compliance, 1+ high issues)
    - Example: Requirements specify 4 STM state machines, but only 3 are configured in XML
    - Example: Required field present but value doesn't match specification
  - ❌ **FAIL**: Critical requirements missing or incorrectly implemented (<90% compliance, 1+ critical issues)
    - Example: DTC specified in requirements but not created in XML
    - Example: Node missing from XML when required by specifications
- **"Skipped" vs "Missing"**:
  - If tracking file says "skipped" or "not found" for a configuration SPECIFIED in requirements, this is a **MISSING configuration** (mark as issue)
  - Do NOT treat database limitations as acceptable excuses - if requirement specifies it, it should be configured
  - Document root cause (e.g., "state machine not in database") but still mark as missing/failed
- **Requirement Source**: Only DEM_Requirements.md determines what is required - if it's documented there, it MUST be in XML
- **Pre-Existing Entities - CRITICAL VERIFICATION PRINCIPLE**: 
  - **Universal Rule:** ANY entity mentioned in DEM_Requirements.md must be verified against requirements, regardless of whether it was newly created or already existed in the database
  - **Applies to ALL entity types:**
    - Failure Words (FW_*)
    - Nodes (Node_*)
    - DTCs (DTC_*)
    - DTC Attributes (DTCAttribute_*)
    - STM Components (STM_PROJECTLINK, STM_NODE)
    - Graph Links (NODE_LINK)
  - **Verification Process for Pre-Existing Entities:**
    1. **Identify:** Look for indicators in tracking files ("already exists", "0 created", "reused existing")
    2. **Locate:** Use grep_search or file_search to find the definition file
    3. **Extract:** Read the entity's full configuration from its definition file
    4. **Compare:** Verify ALL parameters specified in DEM_Requirements.md match the entity's actual configuration
    5. **Report:** Document mismatches as HIGH severity issues
  - **Examples of Pre-Existing Entity Verification:**
    - **Scenario 1:** Workflow says "DTCAttribute_CHR_NonOBD already exists" and requirements specify "Number of Freeze Frame Records: 2"
      - Action: Find DTCAttribute definition file, extract freeze frame records value, compare against requirements
      - If attribute has 3 freeze frames: **Mark as HIGH issue** (mismatch)
    - **Scenario 2:** Workflow creates new DTC linking to existing FW_HydraulicUndervoltage1, and requirements specify FW1 should have "Failed threshold: 100"
      - Action: Find FW definition file (even if not in tracking), verify Failed threshold = 100
      - If FW has threshold 50: **Mark as HIGH issue** (mismatch)
    - **Scenario 3:** Workflow modifies Node_VoltageHandler1 (adds FW references), and requirements specify "Customer: Cus_CHR"
      - Action: Verify node's Customer field matches Cus_CHR
      - If node has Cus_BMW: **Mark as HIGH issue** (mismatch)
  - **Documentation Requirements:**
    - Always document which file contains the pre-existing entity
    - Clearly label in comparison tables whether entity is "New" or "Pre-existing"
    - For pre-existing entities, note the file path of the definition file
    - Track total count of pre-existing entities reviewed
  - **Rationale:** Pre-existing entities may have been created for different use cases with different parameters. Just because they exist doesn't mean they meet current requirements. Full verification is essential for requirements compliance.
