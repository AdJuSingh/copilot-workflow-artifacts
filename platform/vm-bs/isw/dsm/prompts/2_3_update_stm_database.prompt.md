# Update STM Configuration Database - Copilot Prompt

## Prompt Description
This prompt guides the user through updating STM (State Machine) configuration database entries using DiamantPro plugins. It updates STM_PROJECTLINK entries by adding or modifying STM_NODE degradation configurations to specify which nodes should be degraded to which states for each state machine component.

## Usage
Use this prompt when you need to:
- Configure node degradation states for state machines
- Add STM_NODE entries to STM_PROJECTLINKs
- Update existing node degradation configurations
- Define which nodes degrade to specific states when a state machine component is in a particular state
- Ensure proper state machine degradation behavior for diagnostic nodes

## Input File Format

The input file `Diamant_Generate_STMConfig.txt` should follow this format:

```
Node name: Node_VoltageHandler1
Target states: ESPBase = EbdPumpless; AEB = Off; HHC = Off; HBA = Off

Node name: Node_BrakeTemperatureModel
Target states: VLC_AD_Main = Off; CCO_HDC = RampOff

Node name: Node_WssDirFr
Target states: VLC_AD_Main = Off; VLC_CDD_B = Off
```

**Format Rules:**
- Each node configuration starts with `Node name: <NodeName>`
- Followed by `Target states: <STM_REF1> = <State1>; <STM_REF2> = <State2>; ...`
- Multiple target states separated by semicolons (;)
- Each target state in format `<stm_ref> = <state_ref>`
- Blank lines separate different node configurations

## Troubleshooting Guide

### Common Issues

**Issue: "STM_PROJECTLINK not found for stm_ref"**
- Solution: Verify stm_ref name in input file matches database
- Check available STM components: VLC_AD_Main, CCO_HDC, ESPBase, AEB, etc.
- Ensure correct spelling and casing

**Issue: "Node not added/updated"**
- Solution: Check that STM_VAR.STM_NODES collection exists
- Verify node name exists in NODE database
- Check state_ref is valid for that STM component

**Issue: "Input file parsing error"**
- Solution: Verify input file format
- Check "Node name:" prefix exists
- Check "Target states:" prefix exists
- Verify semicolon separators between states
- Verify equals signs between stm_ref and state_ref

**Issue: "Exit code 1 but step succeeded"**
- Solution: **This is normal** - unrelated validation plugins have errors
- Focus on "Total nodes updated/added" output
- Check log file for actual operation results

**Issue: "Plugin ran but changes not in XML file"**
- Cause: WRITESPLITTERS phase failed silently (common on first execution)
- Check for "Splitter [...] written" message in output
- Solution: Re-execute the SAME command - it will succeed on second run
- This is NORMAL behavior when database state needs initialization
- Always verify Node_VoltageHandler1 exists in XML after execution

## Notes
- Environment variable DIAMANT_STM_INPUT_FILE enables CLI mode
- Plugin automatically updates existing nodes or creates new ones
- The `delay_degradation` attribute is always set to "false"
- The `--RELAXED` flag enables permissive validation mode
- The `--WRITESPLITTERS` flag is required to persist changes to XML
- Exit code 1 is normal due to unrelated plugin compilation errors
- STM_PROJECTLINK modifications affect state machine degradation behavior
- **CRITICAL**: First execution may fail to write - always verify "Splitter [...] written" message
- **If write fails**: Re-run immediately with same command - second execution typically succeeds
- **Always verify**: Check XML file contains the node entries after execution
